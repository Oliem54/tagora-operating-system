# Règles de priorisation — Gouvernance TOS

## Objectif

Définir comment prioriser la documentation, l'adoption des standards et les initiatives de convergence vers TOS.

## Portée

Roadmap documentaire TOS, adoption par module et arbitrage des efforts transverses.

---

## Principes de priorisation

1. **Prioriser les standards qui réduisent le risque** — QA, sécurité, validation avant code, séparation des projets.
2. **Prioriser QA, design et validation avant code pour Time** — ordre pilote : TQF → TDS → TES → TKS.
3. **Prioriser la documentation utile avant la documentation parfaite** — un cadre clair et incomplet vaut mieux qu'un traité jamais lu.
4. **Prioriser les décisions réutilisables** — ce qui sert Time et les futurs modules.
5. **Éviter de documenter trop tôt des détails non validés** — pas de valeurs hex inventées, pas de bugs inventés, pas de stack imposée.
6. **Éviter de bloquer l'équipe par excès de processus** — processus léger, exemptions pour correctifs mineurs.

---

## Ordre de référence (pilote Time)

| Priorité | Pilier | Pourquoi |
|---|---|---|
| 1 | **TQF** | Qualité avant production |
| 2 | **TDS** | Signature visuelle commune |
| 3 | **TES** | Méthode avant chaos |
| 4 | **TKS** | Mémoire pour ne pas perdre le fil |
| Ensuite | TAF, TOF, TCP, TBF, TRF, TMF | Selon besoin réel |

---

## Règles

- Fondations d'abord (constitution, piliers, gouvernance).
- Projet pilote TAGORA Time en priorité pour validation terrain.
- Impact écosystème vs effort documenté.
- Pas de sur-documentation prématurée.
- Réviser les priorités à chaque phase de la roadmap documentaire.
- Aligner priorités TOS avec priorités produit des modules.
- Ne pas documenter en détail avant validation du principe.
